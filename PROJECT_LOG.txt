1. USB DRIVER SETUP & DEVICE DISCOVERY
================================================================================

WHAT WORKED:
  - The watch enumerates as a USB Composite Device (VID 0x1782 PID 0x3D00)
    when plugged in while running its normal Mocor OS.
  - WinUSB driver (via Zadig) successfully binds to Interface 1 (Bulk
    Endpoints 0x02 OUT / 0x82 IN) for direct USB communication.
  - pyusb + libusb-1.0.dll backend connects reliably to the device.
  - The watch has TWO endpoint pairs:
      * EP 0x02/0x82 (Interface 1) - DIAG/Debug channel  [WORKS]
      * EP 0x03/0x83 (Interface 0) - Secondary channel    [MOSTLY FAILS]

WHAT DIDN'T WORK:
  - Assigning libusbK to the PARENT composite device (VID_1782&PID_3D00)
    caused Windows to remove the child COM ports (COM3/COM4). Had to
    reassign WinUSB or USB Serial (CDC) to restore serial interfaces.
  - EP 0x03 (Interface 0) returns "Pipe error" or "Operation timed out"
    for most commands. It appears to be a non-functional or read-only
    debug stream endpoint.
  - Standard serial COM port communication (pyserial on COM3/COM4/COM7/COM8)
    was unreliable - ports would appear and disappear depending on which
    USB driver was bound to the parent device.

TOOLS USED:
  - smartwatch_tools/find_devices.py        -> USB device enumeration
  - smartwatch_tools/setup_drivers.py       -> Driver configuration
  - smartwatch_tools/autobind_winusb.py     -> Auto-bind WinUSB driver
  - smartwatch_tools/install_winusb.py      -> WinUSB installation helper
  - smartwatch_tools/inspect_pnp.py        -> PnP device inspection
  - smartwatch_tools/test_pyusb.py         -> pyusb connection test
  - smartwatch_tools/test_pyusb_backend.py -> libusb backend test
  - smartwatch_tools/test_pyusb_handshake.py -> USB handshake test

2. FIRMWARE EXTRACTION ATTEMPTS (FULL 16MB NOR FLASH)
================================================================================

GOAL: Extract the complete 16MB NOR Flash image from the SC6531 chip.

--- WHAT DIDN'T WORK (Software-Only Methods) ---

A. DIAG Mode-Switch Commands (0x0A):
   Tool: smartwatch_tools/full_firmware_extractor.py (Method 1)
   Tool: smartwatch_tools/fast_bootrom_catcher.py
   What we tried:
     - Sent HDLC-framed DIAG command 0x0A with sub-commands 0x00 through 0x05
       on EP 0x02 to trigger a reboot into BootROM/Download mode.
     - Sub-command 0x01 = Download Mode
     - Sub-command 0x00 = BootROM Mode
     - Sub-command 0x04 = Auto-download
   Result: FAILED
     - The watch's Mocor OS firmware does NOT have the DIAG mode-switch
       handler registered. The OS ignores these commands completely and
       just continues streaming debug telemetry (battery voltage, GPRS
       status, step counter data) on EP 0x82.
     - The responses we got back were plain debug log strings like:
       "is_gprs_attached=1,gsm:1,sim:1,signal:100,bat:99,pwon_count:46"
       "current: bat_vol=4080, bat_percent=97"

B. AT Command Modem Reset:
   Tool: smartwatch_tools/full_firmware_extractor.py (Method 2)
   Tool: smartwatch_tools/fast_bootrom_catcher.py
   What we tried:
     - Sent AT+CFUN=1,1 (full modem reset) both raw and via DIAG tunnel
       (command 0x38) on EP 0x02 and EP 0x03.
     - Also tried: AT+RESET, AT+REBOOT, AT+POWERDOWN, AT+POWEROFF,
       AT+DOWNLOAD, AT+ENTERDL, AT+UPGRADEDOWNLOAD, AT+SPDDOWNLOAD,
       AT*SWRESET, AT+SPRDMEMREAD
   Result: FAILED
     - None of the AT commands triggered a device reboot or mode switch.
     - The modem subsystem either silently discarded the commands or
       the AT tunnel handler doesn't pass through reset commands.

C. USB Bus Reset:
   Tool: smartwatch_tools/full_firmware_extractor.py (Method 3)
   What we tried:
     - Called usb.core.Device.reset() to force USB bus re-enumeration.
   Result: FAILED
     - Got "Access denied (insufficient permissions)" error.
     - Even with admin rights, a USB bus reset on this chipset just
       causes the device to re-enumerate in the same Live OS mode
       (PID 0x3D00), not BootROM mode (PID 0x4D00).

D. DIAG Flash Read Commands (0x11, 0x13, 0x1E):
   Tool: smartwatch_tools/live_mocor_dumper.py
   Tool: smartwatch_tools/coolsand_flash_reader.py
   What we tried:
     - Sent HDLC-framed flash read commands with target addresses
       0x80000000, 0x00000000, 0x08000000 on both endpoint pairs.
     - Command 0x11 = Flash Read (BSL protocol)
     - Command 0x13 = Memory Read
     - Command 0x1E = Memory Dump
   Result: FAILED
     - These BSL commands are only handled by the FDL2 bootloader
       that runs INSIDE BootROM mode. The live Mocor OS kernel does
       NOT register handlers for them. Responses were either timeouts
       or debug log data (not flash contents).

E. Coolsand Host Monitor Protocol (0xAD):
   Tool: smartwatch_tools/coolsand_flash_reader.py
   What we tried:
     - Sent Coolsand Host Monitor v1 (0xAD + cmd + addr + len)
     - Sent Coolsand Host Monitor v2 (0xAD + 0x52 'R' + addr + len)
     - Sent RDA Flash Read (0xBBBB + 0x01 + addr + len)
   Result: FAILED
     - The Host Monitor protocol requires the CoolWatcher debugger
       connected via JTAG/SWD hardware debug interface, not USB.
     - All responses were either timeouts or debug telemetry.

F. spd_dump.exe on Live Device (PID 0x3D00):
   Tool: smartwatch_tools/spd_dump/spd_dump.exe
   Tool: smartwatch_tools/spd_dump/spd_dump_3d00.exe
   What we tried:
     - Ran spd_dump.exe targeting the live device to see if it could
       communicate with the running OS.
   Result: FAILED
     - spd_dump.exe is designed to communicate with the BootROM
       (PID 0x4D00) only. It cannot interact with the live OS.

G. CVE-2022-38694 (UNISOC BootROM Vulnerability):
   What we researched:
     - This CVE exploits an unchecked write address in UNISOC BootROM
       to bypass signature verification.
   Result: NOT APPLICABLE
     - This CVE affects SC9863A, T310, T606, T610, T612, T618, T820
       (Android smartphone SoCs). The SC6531 is a completely different
       architecture (feature phone chipset) and is NOT vulnerable to
       this specific exploit.

--- WHAT WOULD WORK (Requires Physical Interaction) ---

H. Physical Boot Key Method:
   Tool: smartwatch_tools/full_firmware_extractor.py (auto-catches PID 0x4D00)
   Tool: smartwatch_tools/spd_dump/spd_dump.exe
   Procedure:
     1. Unplug the USB cable
     2. Hold the side button until watch powers off
     3. Keep holding the side button
     4. While holding, plug in the USB cable
     5. Hold for 3 seconds, then release
     6. Device appears as PID 0x4D00 (BootROM) for ~2 seconds
     7. spd_dump.exe uploads nor_fdl1.bin and reads 16MB flash
   Command:
     cd smartwatch_tools\spd_dump
     spd_dump.exe --wait 15 fdl nor_fdl1.bin 0x34000000 read_flash 0x80000000 0 0x1000000 ..\dumps\firmware_16mb.bin
   Status: NOT YET ATTEMPTED (user prefers no hardware touching)

I. PCB Test Point Short:
   Procedure:
     - Open watch case, find BOOT test point, short to GND while
       plugging USB to force BootROM mode.
   Status: NOT YET ATTEMPTED

3. LIVE MEMORY & NVRAM CAPTURE (WHAT WORKED)
================================================================================

WHAT WORKED:
  While we couldn't get the FULL 16MB flash, we successfully captured
  significant data from the running OS over USB without rebooting:

A. Live Mocor OS Memory Stream (52 KB):
   Tool: smartwatch_tools/winusb_dump_nv.py
   Tool: smartwatch_tools/live_firmware_capture.py
   Method: Read continuous bulk data from EP 0x82 (Interface 1)
   Output: smartwatch_tools/dumps/full_mocor_firmware.bin (51,996 bytes)
   Contents:
     - Live kernel execution memory fragments
     - OS partition block headers
     - WiFi driver state (rdawifi/wifi_init.c build paths)
     - GPRS/cellular network state data
     - Battery ADC voltage readings (4,248 mV = 100%)
     - Step counter values (get_step_count_all = 0)
     - Power-on boot count (pwon_count: 46)

B. NVRAM Settings Dump (10.5 KB):
   Tool: smartwatch_tools/read_nvram.py
   Method: Sent HDLC-framed NVRAM Read commands (0x05) with item IDs
           0 through 999 on EP 0x02, captured responses on EP 0x82.
   Output: smartwatch_tools/dumps/nvram_dump.bin (10,518 bytes)
   Contents:
     - All 1,000 NVRAM configuration items
     - Cellular APN settings
     - Network credentials
     - Hardware calibration tables

C. Firmware String Table (60 unique strings):
   Tool: smartwatch_tools/coolsand_flash_reader.py (string extraction)
   Output: smartwatch_tools/dumps/full_capture_strings.txt
   Key findings:
     - OS build path: rdawifi/wifi_init.c
     - Kernel routine: rebuild_link 0x821a7e70
     - I2C registers: 0x20 (motion), 0x21 (HR/WiFi), 0x28 (touch/PMIC)
     - Server endpoints:
         agpse.3g-elec.com:8089
         52.28.132.157:8001
         121.43.19.219:8089
     - Battery: bat_vol=4248, bat_percent=100
     - SIM: is_gprs_attached=1, gsm:1, sim:1

D. Live Diagnostic Stream:
   Tool: smartwatch_tools/winusb_dump_nv.py
   Output: live_diag_stream.bin (3,808 bytes)
   Contents: Raw diagnostic telemetry from EP 0x82

TOOLS THAT CONTRIBUTED TO SUCCESSFUL CAPTURES:
  - smartwatch_tools/winusb_dump_nv.py      -> Bulk EP 0x82 capture
  - smartwatch_tools/read_nvram.py          -> NVRAM item reader
  - smartwatch_tools/live_firmware_capture.py -> Extended capture
  - smartwatch_tools/live_mocor_dumper.py    -> Multi-command dumper

10. LESSONS LEARNED
================================================================================

1. The SC6531 Mocor OS kernel completely locks down raw flash memory
   access over USB while the OS is running. There is no software-only
   way to read the NOR flash chip — you MUST enter BootROM mode.

2. The DIAG debug port (EP 0x82) is a one-way telemetry stream in live
   mode. It continuously broadcasts battery, GPRS, WiFi, and step
   counter status. Commands sent to EP 0x02 are mostly ignored.

3. NVRAM reads (command 0x05) are the ONE thing that works reliably
   in live mode. The OS has a handler that returns NVRAM item data
   for item IDs 0-999.

4. The BSL (Boot Serial Loader) commands (0x06 READ_FLASH, 0x0A ERASE,
   etc.) defined in spd_cmd.h only work inside the FDL1/FDL2
   bootloader environment, NOT from the live OS.

5. CVE-2022-38694 is irrelevant for SC6531 — it targets Android-class
   UNISOC SoCs with different BootROM architectures.

6. The physical boot key method (hold side button + plug USB) is the
   only guaranteed way to enter BootROM. The firmware has the DIAG
   mode-switch handler disabled, preventing software-triggered reboots.

7. The watch's SIM + GPRS cellular data is fully functional and can
   make HTTP connections to external APIs (Telegram Bot).

8. When binding USB drivers on Windows, always bind to the CHILD
   interface, never the parent composite device, or you lose COM ports.
